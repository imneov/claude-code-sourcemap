# LLM 驱动开发的四大定律：从大模型物理学到工程架构

## 引言：一团混乱

当你用 AI Agent 做软件开发时，你很快会面对一个问题：项目中有大量的信息——PRD、架构文档、部署流程、测试环境、决策记录、编码规范——它们和 Agent 之间到底是什么关系？哪些应该写进 system prompt？哪些应该做成 tool？哪些应该让 Agent 自己去查？

大多数人的做法是凭直觉塞。把能想到的都塞进一个大 CLAUDE.md，或者一个巨大的 system prompt。能跑就行。跑不好就再加几条规则，再塞几段文档。

这就像在牛顿之前造桥。你可以凭经验造出能用的桥，但你不知道它为什么能撑住，也不知道什么时候会塌。你需要的不是更多经验，而是力学定律——从底层告诉你，什么是可能的，什么是必然的，什么是最优的。

这篇文章试图为 LLM 驱动的软件开发建立这样一套"力学定律"。不是工程最佳实践的汇编，不是经验 tips 的集合，而是从大模型的物理本质出发，推导出一套完整的工程架构——并证明这套架构不是"一种选择"，而是在给定约束下的唯一最优解。

## 第一部分：四大定律

### 第零定律：智能的叠加态与坍缩

**一切的起点是一个问题：大模型到底是什么？**

一个训练好的 LLM，在没有接收到任何 prompt 之前，处于一种特殊的状态：它同时"是"所有东西。它可以是诗人，可以是律师，可以是 Kubernetes 专家，可以是量子物理学家。但它不"是"其中任何一个——这些能力以叠加的形式存在，彼此混合、未分化、相互干涉。

**LLM 是智能的叠加态。**

当你给它一段 prompt——"你是一个资深的后端安全工程师"——你做的事情相当于对这个叠加态进行一次"观测"。观测迫使叠加态沿着你指定的方向坍缩（collapse），从模糊的、全能的、低分化的状态，变成一个具体的、锐利的、专注的智能实体。

但坍缩不是免费的。

就像量子力学中的海森堡不确定性原理：你在一个维度上测量得越精确，正交维度上的不确定性就越大。当你让 LLM 沿"安全审计"方向坍缩时，安全相关的推理能力被大幅锐化——它会注意到 SQL 注入、权限逃逸、未验证的输入。但与此同时，它在 UX 设计、架构全局观、创意表达等正交方向上的能力会变弱。

这不是 LLM 的缺陷。这是叠加态坍缩的物理定律。

> **L0：LLM 是智能的叠加态。Prompt 使其沿特定方向坍缩。坍缩增强目标方向的能力，同时衰减正交方向的能力。**

从这条定律直接推导出三个工程结论：

**第一，角色定义是物理必要的。** 不给 Agent 角色，不是"灵活"，而是让它停留在低分化的叠加态。什么都能做一点，什么都不精。角色定义的本质是在智能空间中选择一个坍缩方向，让特定能力从噪声中涌现出来。

**第二，复杂任务必须用多 Agent。** 这不仅是为了并行效率。更根本的原因是：单次坍缩只能锐化一个方向。架构师需要全局视角，安全审计员需要局部穿透力，代码实现者需要工程严谨性——这三个方向在智能空间里几乎正交，单个 Agent 不可能同时在所有方向上保持锐利。多 Agent 架构的本质是允许多次独立坍缩同时存在。

**第三，不同坍缩态必须物理隔离。** 当你在一个 session 里让 Agent 先做架构设计、再做代码实现、再做安全审查，上一次坍缩的残余会"污染"下一次。架构师的全局思维模式会干扰审计员的细节穿透力。这就是为什么成熟的 Agent 框架都提供 `context: fork` 或独立 session 机制——不是为了组织整洁，而是为了保证坍缩的纯度。

### 第一定律：上下文即全部

如果第零定律告诉我们 LLM "是什么"，第一定律开始告诉我们 LLM "怎么工作"。

> **L1：LLM 不拥有知识、记忆或感知。它的全部现实，是 context window 中的 token 序列。**

这不是一个"局限性"。这是这台机器的物理定义。

LLM 没有"忘记"你的项目——它从来就没"知道"过。Context window 之外的一切，对它来说根本不存在。你的 PRD 写了 50 页，如果只有 3 页被加载进 context，LLM 看到的世界就只有那 3 页。另外 47 页，不是"它可能不知道"，而是"不存在"。

而 context window 是有限的。即使是 100 万 token 的窗口，面对一个包含数百个文件的真实项目，也远远不够把所有东西都塞进去。更重要的是，研究表明，随着 context 中信息量的增加，LLM 对每条指令的遵循能力会均匀下降。不是"忽略后面的"，而是"所有的都开始变模糊"。

这意味着：**所有影响 Agent 行为的知识，都必须被显式管理，而且必须精挑细选地加载。**

### 第二定律：每次调用从零开始

> **L2：推理过程无状态。每次调用，LLM 诞生又消亡。调用之间没有任何东西被保留。**

上一轮对话中 Agent 做了一个精妙的架构决策。下一轮调用，那个"Agent"已经不存在了。你面对的是一个全新的实例，对之前发生的一切一无所知——除非你把它重新喂进去。

我们在对话中感受到的"连续性"是一个幻觉。这个幻觉由外部系统维持：对话历史被存储、拼接、重新注入到下一次调用的 context 中。但 LLM 本身没有记忆。它不是"忘了"，它是"从未经历过"。

这条定律的直接推论是：**一切需要跨调用存活的信息，都必须被外部系统显式持久化。** 如果你不把决策记录写到文件里，不把任务状态存到数据库中，不把反馈捕获到日志里——它们就会随着调用的结束永久蒸发。

### 第三定律：输出是概率的，不是确定的

> **L3：LLM 生成的是统计上最可能的 token 序列，不是逻辑上正确的结论。它无法区分"合理"与"正确"。**

传统程序要么对要么错。编译器会告诉你哪一行有 bug。但 LLM 的输出是概率分布的一次采样——它可以流畅地、自信地写出完全错误的代码，而且自己不知道是错的。

这不是说 LLM 不可靠。大多数时候，"统计上最可能"和"正确"是重合的。但它们不是同一件事。LLM 没有内建的验证机制，没有 ground truth 的锚点，没有"我不确定"的可靠自我感知。

这意味着：**LLM 的输出必须经过外部验证，而验证的强度应该与错误的后果成正比。**

## 第二部分：桥接定理——为什么是这套方案

四条定律描述了物理约束。但从约束到具体的工程架构，中间还需要一步推导：为什么这些约束必然导向我们提出的方案，而不是其他方案？

这一步由三条桥接定理完成。每条定理都用反证法证明：任何替代方案都可以被证明是次优的。

### 定理一：选择性加载定理（从 L1 推导）

> 有限上下文 + 异构知识 → 知识必须按任务相关度分层，独立加载。

**反证：** 如果不分层，把所有知识塞进一个大文件会怎样？要么超出 context window（物理不可能），要么挤占推理空间导致指令遵循能力下降。那只加载一部分呢？如果不分层，你无法做到"只加载当前任务需要的那部分"——因为它们是一坨混在一起的文本。

所以知识**必须**被切成独立可加载的单元，并按任务相关度排序加载。这不是"组织整齐比较好"，而是**在有限空间里做最优资源分配的唯一方式**。

从这条定理推导出知识分层：

- **Project Spec**（PRD、架构、数据流）——几乎总是需要，优先加载
- **Decision Log**（ADR、技术选型理由）——做设计决策时加载
- **Conventions**（编码规范、分支策略）——生成代码时加载
- **Task State**（Issue、进度、优先级）——开始任务时加载

### 定理二：频率匹配持久化定理（从 L2 推导）

> 零状态保留 + 异构变更频率 → 每类状态的持久化机制必须匹配其变更节奏。

**反证：** 如果用同一个机制存所有东西会怎样？把 PRD 和测试环境 IP 都放在 Git 里——PRD 没问题，一个迭代改一次。但环境 IP 一天变三次，每次都要 commit-push-pull？反过来，把架构文档放在实时 API 后面——每次 Agent 启动都要查询一个几乎不变的东西？

慢变状态放进快系统 = 不必要的开销。快变状态放进慢系统 = 数据过期。唯一不浪费也不过期的方案，是让持久化机制的读写成本匹配内容的变更频率。

从这条定理推导出持久化栈：

- **Git**（版本化文档）——匹配慢变知识（PRD、架构）
- **Issue Tracker / API**——匹配中等变化（任务状态）
- **MCP / Tool Call**——匹配快变状态（环境信息、CI 状态）

反馈回路也从这里推出：既然 L2 保证了每次调用都从零开始，执行过程中产生的信息如果不被外部系统捕获，就永久消失了。反馈回路是对抗无状态性的唯一手段。

### 定理三：后果比例验证定理（从 L3 推导）

> 概率性输出 + 异构后果严重度 → 验证强度必须与错误代价成正比。

**反证：** 对所有输出做同等验证会怎样？全部自动验证——那改数据库 schema 的代码也自动通过？太危险。全部人工审核——那改个变量名也要人看？太慢。

唯一合理的方案是分级验证：

- **低风险**（格式、语法）→ 自动验证（lint / test / type-check）
- **中风险**（逻辑变更）→ 人工 review
- **高风险**（数据迁移、线上部署）→ 强制人工确认 + 护栏拦截

Human-Agent Protocol 不是流程规范，而是在概率性机器上做风险管理的最优策略。

### 交叉推导：反馈归因定理（L2 × L3）

L2 告诉我们信息不捕获就会消失。L3 告诉我们输出必然有错误。两者结合推出：**执行失败的信息必须被捕获，并且被归因到正确的知识层。**

部署失败了——原因是什么？可能是 Skill 写错了（修 Skill），可能是项目有特殊约束没记录（补 Project Spec），可能是环境变了（刷新 Live State）。如果不做归因就过去了，根据 L2，那条错误信息永久消失——同样的错误一定会重复。

没有归因的反馈不是反馈。它只是噪声。

## 第三部分：完整架构

四条定律 + 三条桥接定理 + 一条交叉推导，共同推导出了一套完整的工程架构。每一层的存在都可以追溯到某条定律的要求，每一层的具体实现方式都可以追溯到某条桥接定理的证明。

### 知识层（从 L0 + L1 + T1 推导）

| 层 | 内容 | 加载时机 | 变更频率 |
|---|---|---|---|
| **Role Definition** | 角色身份、能力方向、行为边界 | 每次坍缩（session 启动） | 按项目/任务类型 |
| **Project Spec** | PRD、架构文档、数据流定义 | 几乎总是加载 | 按迭代周期 |
| **Decision Log** | 技术选型理由、方案取舍、ADR | 做设计决策时按需加载 | 只增不删 |
| **Conventions** | 编码规范、分支策略、安全红线 | 生成代码时加载 | 相对稳定 |
| **Task State** | Issue 列表、优先级、进度、依赖 | 开始任务时加载 | 实时变动 |

这些层的排列顺序不是随意的——它们按照与"当前任务"的相关度从高到低排列，也对应着加载的优先级。当 context 空间紧张时，先牺牲底层（Task State 可以按需查询），保住顶层（Role Definition 和 Project Spec 不能没有）。

### 持久化栈（从 L2 + T2 推导）

| 变更频率 | 存储机制 | 加载方式 | 典型内容 |
|---|---|---|---|
| 慢（周/月级） | Git 版本化文件 | Session 启动时加载 | CLAUDE.md、架构文档、ADR |
| 中（天/任务级） | Issue Tracker API | 任务开始时同步 | Sprint backlog、任务分配 |
| 快（小时/分钟级） | MCP / Tool Call | 按需实时查询 | 环境 IP、CI 状态、部署版本 |

关键原则：不要缓存快变数据，不要频繁查询慢变数据。每一层有自己的节奏，混在一起必然导致过期或浪费。

### 信任层次（从 L3 + T3 推导）

| 验证层级 | 机制 | 触发条件 | 对应风险 |
|---|---|---|---|
| **自动验证** | Lint / Test / Type-check | 每次代码修改后 | 语法错误、类型不匹配 |
| **同行审查** | Review Agent 或人工 review | 逻辑变更、架构修改 | 业务逻辑错误 |
| **强制拦截** | PreToolUse Hook / 权限系统 | 危险操作（删除、部署） | 数据丢失、线上事故 |
| **人工确认** | Human-Agent Protocol | 不可逆变更 | 严重后果 |

验证强度与后果严重度成正比。这不是谨慎，是数学最优。

### 反馈回路（从 L2 × L3 推导）

```
执行结果 → 是否符合预期？
  ├── 是 → 继续
  └── 否 → 归因分析：
        ├── Skill 有 bug → 修改 Skill
        ├── 缺少项目约束 → 补充 Project Spec / Decision Log
        ├── 环境已变化 → 刷新 Live State 查询
        └── 角色不匹配 → 调整 Role Definition / 换 Agent
```

归因是反馈的灵魂。"记住下次不要这样做"不是反馈——"这次失败是因为 deploy skill 没有处理多节点场景，已在 skill 中补充"才是反馈。

## 第四部分：落地——以 Claude Code 为例

理论再漂亮，最终要映射到现有工具上。以 Claude Code 为例，它提供了一套完整的机制栈。我们逐一对照。

### Claude Code 机制清单

**CLAUDE.md**——层级化的上下文文件。`~/.claude/CLAUDE.md` 是全局级别（跨项目），项目根目录的 `CLAUDE.md` 是团队级别（Git 共享），子目录可以有更细粒度的覆写。每次 session 启动时自动加载。

**Skills**——按需加载的能力封装。每个 Skill 是一个 `SKILL.md` 文件，包含元数据（何时触发）和正文（具体指令）。Claude 根据当前任务自动判断是否需要加载某个 Skill——这是一种 LLM 原生的 selective loading。Skill 中的 `!`command`` 语法可以在加载时执行 shell 命令，注入实时数据。

**MCP（Model Context Protocol）**——外部工具连接。通过 `.mcp.json` 配置，Claude 可以连接到 Jira、Slack、Google Drive、数据库等外部系统。这是 Live State 查询的标准通道。

**Hooks**——确定性的拦截点。`PreToolUse`（执行前拦截）、`PostToolUse`（执行后处理）、`PermissionRequest`（权限决策）等。Hook 不是 LLM 决策，是传统代码——100% 确定性，每次都会触发。

**Subagents / Agent Teams**——多 Agent 协作。Subagent 在隔离的 context 中执行（`context: fork`），可以配置不同的模型、权限和系统提示。Agent Teams 允许多个独立 Claude session 并行工作并相互通信。

**权限系统**——工具级别的 allow/deny 规则，支持模式匹配（如 `Bash(git *)` 允许所有 git 命令）。

### 映射与 Gap 分析

**做得好的：**

Conventions 层是最完整的。CLAUDE.md 提供软约束（prompt 级别），Hooks 提供硬约束（确定性代码）。最佳实践已经很明确：不要让 LLM 做 linter 的工作，用 PostToolUse Hook 自动跑 format / lint / type-check。这完美对应了 L3 信任层次中的"自动验证"层。

快变状态的处理也很强。Skills 中的 `!`command`` 是一个精妙的设计——它把 shell 命令的执行结果直接注入到 Skill 的 prompt 中，Claude 只看到处理后的数据。这就是 T2 "按需查询"的工程实现。

**三个关键 Gap：**

**Gap 1：Decision Log 缺失。** Claude Code 没有结构化的决策记录机制。"为什么选了 gRPC 而不是 REST"这种决策，在 session 结束后就消失了。Auto-memory（`#` 快捷方式）只记住片段，没有"为什么"、没有上下文、没有替代方案的对比。这意味着每个新 session 都可能重新踩坑。

**Gap 2：反馈不归因。** Auto-memory 记住的是"别用 X 方法"，但不会告诉你这条教训应该更新到哪里：改 Skill（流程有 bug）？改 CLAUDE.md（项目有特殊约束）？还是改 MCP 查询（环境已变化）？没有归因，反馈只是扁平的文本碎片，无法驱动系统性改进。

**Gap 3：权限模型是工具维度的，不是后果维度的。** 权限系统基于工具类型和命令模式（`Bash(rm *)` vs `Bash(echo *)`），但它无法理解语义风险。"修改一个 README" 和 "修改数据库 migration 文件" 在权限系统看来可能都是 `Write(*.md)` 或 `Write(*.sql)`——但后果天差地别。L3 + T3 要求验证强度与后果成正比，但当前工具只做到了工具粒度，不是后果粒度。

### 角色定义在哪里？

一个重要的发现：**角色定义不是一个新的知识层——它是对现有层的遍历策略。**

当你给 Subagent 定义"你是安全审计工程师"时，你实际上做的是：

- 指定加载哪些 **Conventions**（安全相关的编码规范）
- 关注哪些 **Task State**（安全相关的 Issue）
- 提高 **L3 验证** 的严格度（对权限、输入验证、加密等方面更挑剔）

Agent Teams 中的"reviewer agent"本质上就是 L3 验证层的一个实现——让另一个 Agent（独立坍缩态）做 consequence-scaled verification。

角色 ≠ 新层。角色 = 一套策略，决定如何遍历现有的知识层、持久化栈和信任层次。

## 第五部分：交叉张力与工程权衡

四条定律之间不是孤立的。它们的交叉组合产生了三组重要的工程权衡，任何实际系统都必须面对。

### L0 × L1：锐度-容量权衡

坍缩需要 prompt，知识需要 context，两者共享同一个有限空间。角色定义越详细（"你是一个专注于 Kubernetes 网络策略的、有 10 年经验的安全工程师，偏好声明式配置，反对过度抽象……"），坍缩越精确，但留给项目知识的空间就越少。

这个权衡没有通用的最优解，但有一个原则：**角色定义应该足够锐利以区分方向，但不要锐利到挤占关键的项目上下文。** 在实践中，这意味着 50-200 词的角色定义 + 结构化的项目知识，通常优于 2000 词的超详细人设 + 残缺的项目信息。

### L0 × L2：坍缩的瞬时性

每次调用，坍缩都要重新发生。上一轮 session 中 Agent 已经完美地"成为"了一个架构师——下一轮，那个坍缩态消失了。你必须重新注入 prompt 让它再坍缩一次。角色不会被"记住"，只会被"重建"。

这就是为什么 CLAUDE.md 和 Subagent 配置文件必须是持久化的、版本化的——它们不是一次性的 prompt，而是坍缩的模板，需要在每次调用时可靠地重现同一个智能实体。

### L0 × L3：坍缩不等于确定

即使 prompt 精确到"你是专注于 Kubernetes NetworkPolicy 审计的安全工程师"，输出仍然是概率性采样。坍缩只是移动了概率分布的中心，让安全相关的推理更可能出现——但它没有消除方差。Agent 仍然可能遗漏一个明显的 CVE，仍然可能写出"看起来安全但实际有漏洞"的代码。

这就是为什么即使有完美的角色定义，你仍然需要 L3 的验证层。坍缩和验证是互补的，不是替代的。坍缩提高了"基准线"（大部分输出更好），验证捕获"尾部风险"（偶尔的严重错误）。

## 结语

回到最初的问题：Agent 开发中，那些散落的 PRD、架构文档、部署流程、测试环境信息，和 Agent 到底是什么关系？

现在我们可以给出一个完整的回答：

**它们是四条定律在工程层面的具体投射。** PRD 和架构文档是 Project Spec 层（L1 要求显式管理）。部署流程和 CICD 定义是 Skill（L1 的可复用能力封装）。测试环境信息是 Live State（L2 + T2 要求实时查询）。代码规范和安全红线是 Conventions（L3 + T3 的软硬约束）。角色定义是 L0 的坍缩方向。反馈记录是 L2 × L3 的归因产物。

每一层的存在都不是"组织整齐比较好"，而是"不这样做会违反某条定律"。

整套架构可以浓缩为四个问题：

- **L0：让它成为什么？** → 角色定义、多 Agent、坍缩隔离
- **L1：喂它什么？** → 知识分层、选择性加载
- **L2：留住什么？** → 频率匹配持久化、反馈回路
- **L3：信任什么？** → 分级验证、后果比例升级

**四条定律。四个问题。一套架构。**

不是偏好。不是经验。是推导。
