# 代码开发的认知结构与 LLM 驱动开发的物理定律

## ——从第一性原理推导人机协作的工程架构

---

## 前言

当你用 AI Agent 写代码时，你很快会面对一个问题：项目中有大量的信息——PRD、架构文档、部署流程、测试环境、决策记录、编码规范——它们和 Agent 之间到底是什么关系？哪些应该写进 system prompt？哪些应该做成 tool？哪些应该让 Agent 自己去查？

大多数人的做法是凭直觉塞。把能想到的都塞进一个大 CLAUDE.md，或者一个巨大的 system prompt。能跑就行。跑不好就再加几条规则，再塞几段文档。

这就像在牛顿之前造桥。你可以凭经验造出能用的桥，但你不知道它为什么能撑住，也不知道什么时候会塌。你需要的不是更多经验，而是力学定律——从底层告诉你，什么是可能的，什么是必然的，什么是最优的。

但在建立"力学定律"之前，我们还需要回答一个更基本的问题：**代码开发这件事本身，到底在做什么？** 不是从组织管理的角度（那是瀑布和敏捷回答的问题），而是从认知活动的角度——当一个人（或一个 Agent）在"开发软件"时，他的大脑（或推理引擎）经历了哪些不可再分的阶段？

只有搞清楚这个问题，我们才能精确地回答：哪些阶段可以交给 LLM，哪些不能，边界在哪里，接口是什么。

这篇文章试图建立两层理论，并推导出它们的交汇点：

- **上层：代码开发的认知模型** ——四个不可再分的阶段，描述"写代码这件事的本质结构"
- **下层：LLM 驱动开发的四条定律** ——四条物理约束，描述"当你把部分阶段委托给 LLM 时，这台机器的工作原理"
- **接口：人机边界** ——两层之间的精确连接点，决定了整个系统的天花板

最终，从这套理论中，我们将推导出一条完整的信息流管线，识别出当前工具链中的关键缺口，并指向一个具体的工具设计方向。

# 第一部分：代码开发的认知结构

## 第一章：剥掉组织形式的外壳

软件工程的历史，主要是组织形式的历史。瀑布模型回答的是"怎么安排阶段和交付物"。敏捷回答的是"怎么缩短反馈周期"。DevOps 回答的是"怎么打通开发和运维"。它们都是在回答"怎么组织人和时间"。

但它们都没有回答一个更底层的问题：**写代码这件事本身，到底在做什么？**

如果我们剥掉所有组织形式的外壳，回到认知活动本身，代码开发可以被还原为四个不可再分的阶段。这四个阶段不是某种方法论的主张，而是对"将意图转化为可执行指令"这个过程的认知分析。无论你用瀑布还是敏捷，无论你是一个人还是一百人的团队，无论你用 Vim 还是 AI Agent，这四个阶段都存在。

## 第二章：四个认知阶段

### 理解：从模糊到清晰

这个阶段的本质是在问题空间里建立心智模型。

不管你是在读需求文档、和用户对话、还是盯着一段遗留代码试图搞懂它在做什么，你做的都是同一件事：把外部世界的复杂性内化为你头脑中可操作的结构。

理解不是一次性完成的。你跟客户聊了一次，形成了一个初步认知，然后去现场看了实际业务流程，发现认知有偏差，回来修正。这个循环可能跨越几天甚至几周，中间涉及的信息源是非结构化的、多模态的、高度依赖领域经验的。

这个阶段的产出不是文档——虽然它通常会被编码为文档（PRD、需求规格说明）。它的真正产出是"你真正理解了什么"。文档只是理解的投影，不是理解本身。

### 决策：从可能到确定

理解了问题之后，方案空间是开放的。

用什么数据结构、怎么划分模块边界、接口长什么样、用现成的库还是自己写、选 gRPC 还是 REST、数据库放在边缘还是云端——每一步都是在一个巨大的可能性树上做剪枝。

架构设计是宏观决策，变量命名是微观决策，但本质相同：在约束条件下选择一条路径。

这个阶段最容易被低估，因为很多决策是隐性的——写代码的人自己都没意识到自己在做决策。"用 Go 写这个服务"是一个决策。"不做这个优化"也是一个决策。"先不处理这个边界情况"还是一个决策。

决策的关键特征是：它依赖的信息散落在多个来源中（性能测试结果、团队技能储备、上下游系统的约束、甚至组织政治），这些信息的变更频率和获取方式完全不同，不可能在一次集中处理中凑齐。决策过程的反馈循环往往很长——你选了一个方案，可能要到上线后才知道选对了没有。

### 表达：从意图到形式

这是唯一真正在"写代码"的阶段。

你把前两个阶段的认知成果翻译成编程语言能理解的形式化指令。这个阶段的核心挑战是保真度——你脑子里的模型和你写出来的代码之间，总会有信息损耗。类型系统、命名规范、注释，本质上都是在对抗这种损耗。

表达阶段有一个重要的内部结构：它不是一步到位的。一个前端工程师写完一段代码，刷新浏览器看效果，发现按钮位置不对，回来改代码，再刷新——这个"生成→观测→修正"的紧循环，是表达阶段的内部行为，不是独立的验证。认知主体没变，意图没变，只是形式化的过程在迭代。

### 验证：从假设到事实

你写的每一行代码都隐含一个假设："这段逻辑能正确处理这种情况。"验证就是把假设变成事实，或者把错误暴露出来。

但真正的验证有一个关键要求：**它必须由独立的视角完成。**

写代码的人自己检查自己的代码，能发现一些表面错误（语法、类型），但很难发现深层问题——因为他的认知已经被"我认为这是对的"锁定了。这就像校对自己的文章：你的眼睛会自动补全你脑中的意思，跳过实际的错误。

所以验证阶段的核心不是"检查"这个动作，而是"独立视角"这个条件。Code review 是验证，因为审查者有不同的心智模型。安全审计是验证，因为审计者关注的维度和开发者不同。而"跑了一下测试发现没报错"——这取决于测试是谁写的。如果是同一个人写的代码和测试，测试很可能和代码有同样的盲区。

## 第三章：四个阶段的关系

这四个阶段不是线性的。它们在每一个粒度上都在循环。

你可能在写一个函数的过程中，在几分钟内就完成了理解→决策→表达→验证的完整循环。你可能在一个项目的层面上，用几个月完成一次大循环。瀑布模型试图在宏观上把它们拉成直线，敏捷试图缩短循环周期，但**循环本身是不可消除的**。

关键的洞察是：虽然四个阶段都存在，但它们的认知特征截然不同：

- **理解**和**决策**的反馈循环长、信息源分散、高度依赖领域经验和隐性知识。它们的认知主体必须是能够在长时间跨度上积累和修正认知的实体。
- **表达**和**验证**的反馈循环短、输入输出可以被结构化、规则性强。它们的核心挑战是保真度和覆盖度，而不是深层理解。

这个分野，直接指向了人和 LLM 的分工边界。

# 第二部分：LLM 的物理定律

## 第四章：为什么需要"定律"

上一部分描述了代码开发的认知结构——四个阶段是普适的，跟你用不用 AI 无关。这一部分进入 LLM 的世界，回答的问题是：当你决定把某些阶段委托给 LLM 时，这台机器有什么物理约束？

大多数关于 AI 编码的文章，给出的是"最佳实践"——用 CLAUDE.md、分 Subagent、加 Hook。但它们不解释为什么这些做法不是偏好而是必要。这就像告诉你"桥的跨度不要超过 100 米"，但不告诉你万有引力和材料力学——你不知道什么时候可以突破 100 米，什么时候 50 米也会塌。

我们需要的是一组定律：从 LLM 的基本特性出发，推导出工程约束，再从约束推导出架构。这样得到的架构不是"一种选择"，而是在给定条件下的唯一最优解。

## 第五章：四条定律

### L0：智能的叠加态与坍缩

**一切的起点是一个问题：大模型到底是什么？**

一个训练好的 LLM，在没有接收到任何 prompt 之前，处于一种特殊的状态：它同时"是"所有东西。它可以是诗人，可以是律师，可以是 Kubernetes 专家，可以是量子物理学家。但它不"是"其中任何一个——这些能力以叠加的形式存在，彼此混合、未分化、相互干涉。

**LLM 是智能的叠加态。**

当你给它一段 prompt——"你是一个资深的后端安全工程师"——你做的事情相当于对这个叠加态进行一次"观测"。观测迫使叠加态沿着你指定的方向坍缩，从模糊的、全能的、低分化的状态，变成一个具体的、锐利的、专注的智能实体。

但坍缩不是免费的。当你让 LLM 沿"安全审计"方向坍缩时，安全相关的推理能力被大幅锐化——它会注意到 SQL 注入、权限逃逸、未验证的输入。但与此同时，它在 UX 设计、架构全局观、创意表达等正交方向上的能力会变弱。

> **L0：LLM 是智能的叠加态。Prompt 使其沿特定方向坍缩。坍缩增强目标方向的能力，同时衰减正交方向的能力。**

![图二：L0 叠加态与坍缩——prompt 锐化一个方向，同时衰减其他方向](figures/fig2_l0_superposition_collapse.svg)

**关于类比的边界：** 需要诚实地声明，L0 使用的是量子力学的类比，而非严格推导。量子力学中的不确定性原理是数学上可证明的硬约束，而 LLM 中"角色坍缩会衰减正交能力"目前更多是工程经验的系统性观察。我们选择这个类比，是因为它在直觉上精确地捕捉了 LLM 的行为特征，并且能够正确地推导出工程结论。但读者应该意识到，这里的"物理定律"是工程层面的定律（类似热力学定律从宏观现象中总结），而非从 Transformer 数学结构严格推导的定理。

从 L0 直接推导出三个工程结论：

**第一，角色定义是物理必要的。** 不给 Agent 角色，不是"灵活"，而是让它停留在低分化的叠加态。什么都能做一点，什么都不精。角色定义的本质是在智能空间中选择一个坍缩方向，让特定能力从噪声中涌现出来。

**第二，复杂任务必须用多 Agent。** 这不仅是为了并行效率。更根本的原因是：单次坍缩只能锐化一个方向。架构师需要全局视角，安全审计员需要局部穿透力，代码实现者需要工程严谨性——这三个方向在智能空间里几乎正交，单个 Agent 不可能同时在所有方向上保持锐利。多 Agent 架构的本质是允许多次独立坍缩同时存在。

**第三，不同坍缩态必须物理隔离。** 当你在一个 session 里让 Agent 先做架构设计、再做代码实现、再做安全审查，上一次坍缩的残余会"污染"下一次。架构师的全局思维模式会干扰审计员的细节穿透力。这就是为什么成熟的 Agent 框架都提供独立 session 机制——不是为了组织整洁，而是为了保证坍缩的纯度。

### L1：上下文即全部

如果 L0 告诉我们 LLM "是什么"，L1 开始告诉我们 LLM "怎么工作"。

> **L1：LLM 不拥有知识、记忆或感知。它的全部现实，是 context window 中的 token 序列。**

这不是一个"局限性"。这是这台机器的物理定义。

LLM 没有"忘记"你的项目——它从来就没"知道"过。Context window 之外的一切，对它来说根本不存在。你的 PRD 写了 50 页，如果只有 3 页被加载进 context，LLM 看到的世界就只有那 3 页。另外 47 页，不是"它可能不知道"，而是**不存在**。

而 context window 是有限的。即使是 100 万 token 的窗口，面对一个包含数百个文件的真实项目，也远远不够把所有东西都塞进去。更重要的是，研究表明，随着 context 中信息量的增加，LLM 对每条指令的遵循能力会均匀下降。不是"忽略后面的"，而是"所有的都开始变模糊"。

这意味着：**所有影响 Agent 行为的知识，都必须被显式管理，而且必须精挑细选地加载。**

### L2：每次调用从零开始

> **L2：推理过程无状态。每次调用，LLM 诞生又消亡。调用之间没有任何东西被保留。**

上一轮对话中 Agent 做了一个精妙的架构决策。下一轮调用，那个"Agent"已经不存在了。你面对的是一个全新的实例，对之前发生的一切一无所知——除非你把它重新喂进去。

我们在对话中感受到的"连续性"是一个幻觉。这个幻觉由外部系统维持：对话历史被存储、拼接、重新注入到下一次调用的 context 中。但 LLM 本身没有记忆。它不是"忘了"，它是"从未经历过"。

这条定律的直接推论是：**一切需要跨调用存活的信息，都必须被外部系统显式持久化。** 如果你不把决策记录写到文件里，不把任务状态存到数据库中，不把反馈捕获到日志里——它们就会随着调用的结束永久蒸发。

### L3：输出是概率的，不是确定的

> **L3：LLM 生成的是统计上最可能的 token 序列，不是逻辑上正确的结论。它无法区分"合理"与"正确"。**

传统程序要么对要么错。编译器会告诉你哪一行有 bug。但 LLM 的输出是概率分布的一次采样——它可以流畅地、自信地写出完全错误的代码，而且自己不知道是错的。

这不是说 LLM 不可靠。大多数时候，"统计上最可能"和"正确"是重合的。但它们不是同一件事。LLM 没有内建的验证机制，没有 ground truth 的锚点，没有"我不确定"的可靠自我感知。

这意味着：**LLM 的输出必须经过外部验证，而验证的强度应该与错误的后果成正比。**

# 第三部分：两层结构与人机边界

## 第六章：两层结构的咬合

现在我们有了两层理论：

- **上层（认知模型）：** 理解 → 决策 → 表达 → 验证。描述代码开发本身。
- **下层（物理定律）：** L0（坍缩）、L1（上下文）、L2（无状态）、L3（概率性）。描述 LLM 的约束。

它们之间是什么关系？

关键洞察是：**四条定律不是在描述"怎么用 LLM 做软件开发"，而是在描述"怎么用 LLM 做软件开发中它能做好的那部分"。**

理解和决策不在定律的管辖范围内，不是因为遗漏，而是因为这两个阶段的认知特征与 LLM 的物理约束从根本上冲突：

- **理解的反馈循环太长。** 你跟客户聊、去现场看、回来修正认知，这个循环可能跨越数周。按 L2，每次调用从零开始——LLM 根本无法在这种时间跨度上积累和修正认知。
- **理解的信息源无法被 token 化。** 按 L1，LLM 的全部现实是 context window 中的 token。但理解阶段涉及的信息——客户的语气、现场的物理环境、行业的隐性规则——很多根本无法被编码为 token 序列。
- **决策依赖的上下文无法在单次加载中凑齐。** "选 gRPC 还是 REST"的真正决策依据散落在性能测试结果、团队技能储备、上下游系统约束、甚至组织政治中。按 L1，这些东西的变更频率和获取方式完全不同，不可能一次性塞进 context。

而表达和验证恰恰是 LLM 擅长的领域：

- **表达的反馈循环短。** 写代码、跑测试、看结果、改代码——这个循环可以在秒级完成，完全在一次调用的生命周期内。
- **表达的输入可以被结构化。** 任务描述、编码规范、API 文档——这些是 LLM 最擅长消费的结构化 token 序列。
- **验证的规则可以被形式化。** Lint 规则、类型检查、测试用例——这些是确定性的、可自动化的。

所以两层结构的咬合点是精确的：

- **人类负责理解和决策，** 产出知识产物（PRD、ADR、架构文档、约束清单）
- **LLM 负责表达和验证，** 消费知识产物，产出代码和验证结果
- **两层之间的接口，** 就是那些知识产物——人类认知的编码形式，LLM 的输入材料

## 第七章：人机边界——系统的瓶颈

这条边界是整个系统中最关键的一环。

人类在理解和决策阶段的工作，必须在这条边界上被转化为"LLM 可消费的知识产物"。如果转化不完整——理解了但没写成 PRD，做了决策但没记录成 ADR——那么 LLM 在表达阶段就缺少关键输入，产出的代码就会偏离意图。

**不是 LLM 不够聪明，是你没有喂够。** 这是 L1 的直接推论：context window 之外的一切不存在。你脑子里有完美的架构构想，但如果它只停留在你脑子里，对 LLM 来说等于不存在。

这意味着，整个系统的天花板不是由 LLM 的能力决定的，而是由人机边界上知识产物的质量决定的。更具体地说：

- **PRD 的完整度，** 决定了 LLM 能否理解"要做什么"
- **ADR 的存在与否，** 决定了 LLM 能否理解"为什么这样做"
- **约束清单的精确度，** 决定了 LLM 能否理解"什么不能做"
- **任务拆分的粒度，** 决定了 LLM 能否在单次调用中完成一个有意义的单元

这是一个反直觉的结论：**提升 AI 编码效率的最高杠杆，不是更好的模型或更长的 context，而是更好的人类文档。** 这听起来老套，但它是从第一性原理推导出来的——不是"文档很重要"这种泛泛的说教，而是"L1 要求所有知识必须显式加载，所以没被编码的知识等于不存在"这样的逻辑必然。

## 第八章：LLM 在人类阶段的辅助角色

需要澄清一个可能的误解：说"理解和决策是人类的工作"，不意味着 LLM 在这两个阶段完全没用。

"把这 20 个 API 的文档读一遍，告诉我哪些支持流式传输"——这发生在理解阶段，但它不是 LLM 在"理解"。它是 LLM 在执行一个信息检索任务，产出物喂给人类，由人类来理解。认知主体没变，只是工具变了。

类似的辅助角色还包括：整理会议纪要、对比竞品功能、汇总技术方案的 trade-off。这些都是独立的、短循环的、输入输出可结构化的任务——它们本身符合表达阶段的特征，只不过服务于人类的理解和决策过程。

这个区分很重要，因为它防止了一个危险的滑坡：认为"既然 LLM 能帮忙查资料，那它也能帮忙理解需求"。能帮忙查资料和能理解需求之间的距离，就像能帮忙查字典和能读懂一首诗之间的距离。

# 第四部分：桥接定理——从定律到架构

## 第九章：为什么需要桥接定理

四条定律描述了物理约束。但从约束到具体的工程架构，中间还需要一步推导：为什么这些约束必然导向特定的方案，而不是其他方案？

这一步由三条桥接定理完成。每条定理都可以用反证法验证：任何替代方案都可以被证明是次优的。

## 第十章：三条桥接定理

### 定理一：选择性加载定理（从 L1 推导）

> 有限上下文 + 异构知识 → 知识必须按任务相关度分层，独立加载。

**反证：** 如果不分层，把所有知识塞进一个大文件会怎样？要么超出 context window（物理不可能），要么挤占推理空间导致指令遵循能力下降。那只加载一部分呢？如果不分层，你无法做到"只加载当前任务需要的那部分"——因为它们是一坨混在一起的文本。

所以知识**必须**被切成独立可加载的单元，并按任务相关度排序加载。这不是"组织整齐比较好"，而是在有限空间里做最优资源分配的唯一方式。

从这条定理推导出知识分层：

- **Role Definition**（角色身份、能力边界）——每次坍缩时加载
- **Project Spec**（PRD、架构、数据流）——几乎总是需要，优先加载
- **Decision Log**（ADR、技术选型理由）——做设计决策时加载
- **Conventions**（编码规范、分支策略）——生成代码时加载
- **Task State**（Issue、进度、优先级）——开始任务时加载

这些层的排列顺序不是随意的——它们按照与"当前任务"的相关度从高到低排列，也对应着加载的优先级。当 context 空间紧张时，先牺牲底层（Task State 可以按需查询），保住顶层（Role Definition 和 Project Spec 不能没有）。

### 定理二：频率匹配持久化定理（从 L2 推导）

> 零状态保留 + 异构变更频率 → 每类状态的持久化机制必须匹配其变更节奏。

**反证：** 如果用同一个机制存所有东西会怎样？把 PRD 和测试环境 IP 都放在 Git 里——PRD 没问题，一个迭代改一次。但环境 IP 一天变三次，每次都要 commit-push-pull？反过来，把架构文档放在实时 API 后面——每次 Agent 启动都要查询一个几乎不变的东西？

慢变状态放进快系统 = 不必要的开销。快变状态放进慢系统 = 数据过期。唯一不浪费也不过期的方案，是让持久化机制的读写成本匹配内容的变更频率。

从这条定理推导出持久化栈：

- **Git（版本化文档）** ——匹配慢变知识（PRD、架构、ADR），按周/月变更
- **Issue Tracker / API** ——匹配中等变化（任务状态），按天/任务变更
- **MCP / Tool Call** ——匹配快变状态（环境信息、CI 状态），按小时/分钟变更

![图五：知识分层与持久化匹配——每一层知识对应频率匹配的存储机制](figures/fig5_knowledge_layers_persistence.svg)

### 定理三：后果比例验证定理（从 L3 推导）

> 概率性输出 + 异构后果严重度 → 验证强度必须与错误代价成正比。

**反证：** 对所有输出做同等验证会怎样？全部自动验证——那改数据库 schema 的代码也自动通过？太危险。全部人工审核——那改个变量名也要人看？太慢。

唯一合理的方案是分级验证：

- **低风险**（格式、语法）→ 自动验证（lint / test / type-check）
- **中风险**（逻辑变更）→ 独立 Agent 审查
- **高风险**（数据迁移、线上部署）→ 强制人工确认 + 护栏拦截

### 交叉推导：反馈归因定理（L2 × L3）

L2 告诉我们信息不捕获就会消失。L3 告诉我们输出必然有错误。两者结合推出：**执行失败的信息必须被捕获，并且被归因到正确的知识层。**

部署失败了——原因是什么？可能是 Skill 写错了（修 Skill），可能是项目有特殊约束没记录（补 Project Spec），可能是环境变了（刷新 Live State）。如果不做归因就过去了，根据 L2，那条错误信息永久消失——同样的错误一定会重复。

**没有归因的反馈不是反馈。它只是噪声。**

# 第五部分：完整的信息流管线

## 第十一章：管线总览

四条定律 + 三条桥接定理 + 两层认知模型，共同推导出一条完整的信息流管线。这条管线从业务场景出发，经过理解、决策、表达、验证四个阶段，最终通过反馈归因形成闭环。

管线中有一条关键的分界线——人机边界。边界以上，认知主体是人，四条定律不适用。边界以下，认知主体是 LLM，四条定律全面生效。

以下是管线的逐节点规格。

## 第十二章：理解阶段——全人类

理解阶段包含三个信息流节点：

**节点 1：业务场景 → 问题定义。** 输入是非结构化的——客户对话、现场观察、行业报告、竞品分析。产出是结构化的问题陈述：这个系统要解决什么问题，不解决什么问题，边界在哪里。这个节点的信息损耗最大，因为它是从非结构化到结构化的第一次转换。

**节点 2：问题定义 → 约束清单。** 输入是问题定义加上现有系统的技术现状。产出是两类约束：硬约束（必须跑在 ARM 上、延迟不能超过 50ms、数据不能出境）和软约束（团队没有 Rust 经验、预算有限、时间紧迫）。

**节点 3：问题定义 + 约束 → PRD。** 前两个节点的产出被整合、编码成一个 LLM 可消费的文档。这是理解阶段的最终编码产出。注意：PRD 不是理解本身，它是理解的投影——但它是唯一能跨过人机边界的形式。

LLM 在这个阶段可以做的辅助工作：查询 API 文档、整理会议纪要、对比竞品功能、汇总行业报告。但这些都是独立的信息检索任务——它们的产出是原材料，喂给人类，由人类完成真正的"理解"。

## 第十三章：决策阶段——全人类

决策阶段包含三个信息流节点：

**节点 4：PRD + 技术全景 → 候选方案。** 在约束条件下枚举可能的架构路径。这个节点的输出通常是 2-5 个可行方案的粗略描述。

**节点 5：候选方案 + 评估标准 → ADR。** 选定方案并记录完整的决策上下文：选了什么，为什么选它，为什么不选其他的，什么条件下需要重新评估。ADR（Architecture Decision Record）是决策阶段最关键的编码产出。

**节点 6：ADR + PRD → 模块定义 + 任务清单。** 将架构决策细化为可执行的模块划分和任务列表。每个任务应该是 LLM 在单次调用中可以完成的粒度。

这里是整条管线中最大的缺口之一：**ADR 在绝大多数团队中不存在。** 决策在会议中发生，在脑子里完成，在 Slack 里讨论几句就过去了，从未被编码为持久化的、结构化的文档。这意味着跨过人机边界后，LLM 拿到的是残缺的上下文——它知道要做什么（任务），知道怎么写（规范），但不知道为什么这样做（决策理由）。

没有"为什么"的代码是脆弱的。当环境变化时，LLM 不知道哪些实现细节是偶然的（可以改），哪些是有意的（改了就坏）。

## 第十四章：人机边界——知识的格式转换

理解和决策阶段的产出——PRD、ADR、约束清单、模块定义、任务清单——在人机边界上必须完成一次格式转换：从"人类的认知产物"变成"LLM 可消费的知识单元"。

这个转换有三个要求：

**结构化。** LLM 不擅长从长篇散文中提取关键信息。PRD 不应该是一篇叙事性的文章，而应该是分层的、标签化的、可以被选择性加载的结构化文档。

**分层。** 按照定理一的要求，不同类型的知识应该被切成独立可加载的单元。PRD 和 ADR 不应该混在同一个文件里——它们的加载时机不同（PRD 几乎总是需要，ADR 只在设计决策时需要）。

**完整性可检查。** 在人机边界上应该有一道完整性检查：有 PRD 但没 ADR？有任务清单但没有模块定义？缺少约束清单？这些缺失应该被检测并提醒。

当前没有工具系统性地解决这个问题。大多数开发者凭经验判断"够了就行"。这不是懒惰，是因为缺少一个框架告诉他们"什么算够"。我们的框架给出了明确的答案：**PRD + ADR + 约束清单 + 模块定义 + 任务清单，五个知识产物缺一不可。**

## 第十五章：表达阶段——LLM 主导，含自校正循环

表达阶段是 LLM 的主场。四条定律在这里全面生效。

**输入。** 按照 L1 和定理一的要求，表达阶段的输入是选择性加载的知识组合：当前任务 + 相关 ADR + 编码规范 + 必要的项目上下文。L0 控制坍缩方向——根据任务类型选择合适的角色定义（前端开发者、后端工程师、数据库管理员）。

**自校正循环。** 这是对管线的一个关键修正。表达阶段内部包含一个紧循环：

```
生成代码 → 观测结果 → 修正 → 再生成
```

"观测"的手段包括：跑测试看是否通过、用浏览器看渲染效果、检查 Lint 输出、读编译器报错。这个循环的特征是：

- 认知主体没变（同一个 Agent，同一个坍缩态）
- 意图没变（实现同一个任务）
- 反馈循环极短（秒级）

所以这不是"验证"——这是表达过程中的自我修正，就像一个画家边画边退后看整体效果，再回来调整细节。他没有在做"审查"，他在做"表达"。

随着工具能力的增强——Agent 能用浏览器、能跑测试、能读日志、能查数据库——这个自校正循环的能力会越来越强。这是当前 AI 编码工具进步最快的领域。

![图四：表达阶段的自校正循环 vs 验证阶段的独立坍缩——两者的本质区别是认知主体是否相同](figures/fig4_expression_loop_vs_verification.svg)

**产出。** 代码、文档、Schema、配置文件。按 L2 的要求，所有产出必须被持久化——写到文件系统、提交到 Git、记录到 Issue Tracker。Agent 的一次调用结束后，如果产出没有被外部系统捕获，就永久消失了。

**退出条件。** Agent 说"我做完了"的那个时刻，标志着表达阶段的结束。这个时刻是管线中的一个关键触发点——它应该自动触发下一阶段（独立验证）。

## 第十六章：验证阶段——独立坍缩态

验证阶段的核心不是"检查"这个动作，而是"独立视角"这个条件。它和表达阶段的自校正循环有一个根本区别：**做验证的实体和做实现的实体，不能是同一次坍缩。**

这是 L0 的直接要求。写代码的 Agent 沿"实现"方向坍缩，它能看到测试是否通过、渲染是否正确，但它看不到两件事：

1. **实现是否符合业务意图。** 代码在技术上是正确的，但可能完全不是需求要的。这需要一个沿"业务理解"方向坍缩的视角。
2. **实现是否引入了安全漏洞。** 代码功能正常，但可能有 SQL 注入、权限逃逸。这需要一个沿"安全审计"方向坍缩的视角。

验证阶段包含三个递进的节点，验证强度与后果严重度成正比（L3 + 定理三）：

**节点 7：逻辑审查。** 由一个独立的 Review Agent（或人类）完成。这个 Agent 有自己的坍缩方向（侧重于业务逻辑和架构一致性），有自己独立的 context（不被实现 Agent 的思路"污染"）。它问的核心问题是："这段代码做的事情，是需求真正要的吗？"

**节点 8：安全审计。** 由一个专门的安全角色（Agent 或人类）完成。沿"安全"方向深度坍缩，关注权限、输入验证、加密、依赖漏洞。这个角色看到的东西和实现者看到的完全不同——同一段代码，实现者看到的是"功能完成了"，安全审计者看到的是"这里有一个未验证的输入"。

**节点 9：人工确认。** 不可逆操作（数据库 migration、线上部署、删除操作）的最后一道门。这不是对 LLM 的不信任，而是对概率性系统的风险管理——L3 保证了输出永远有非零的错误概率，当错误后果足够严重时，概率再低也值得人类确认。

## 第十七章：反馈归因——闭合循环

管线的最后一个组件是反馈归因。它不是一个独立的"阶段"，而是一个跨阶段的闭环机制。

当验证阶段发现问题时——逻辑审查不通过、安全审计发现漏洞、人工确认被拒绝——问题本身已经被识别了。但如果信息到此为止，根据 L2，它会在下一次调用时永久消失，同样的错误一定会重复。

反馈归因的核心是：**不仅要记住"出了什么问题"，还要判断"问题应该更新到哪里"。**

归因的目标有四个：

- **修 Skill。** 如果问题是执行流程本身有 bug（比如 deploy skill 没有处理多节点场景），修改 Skill 文件，让所有后续执行都避免这个问题。
- **补 ADR / Spec。** 如果问题是缺少项目约束或决策记录（比如"这个接口之所以用 POST 而不是 GET，是因为有防火墙限制"），补充到对应的知识层，让后续 Agent 能加载这个信息。
- **刷新状态。** 如果问题是环境已经变化（测试环境 IP 换了、依赖版本升级了），刷新 Live State 查询机制。
- **调整角色。** 如果问题是角色定义不匹配（比如用"前端开发者"角色做了一个需要"全栈工程师"视角的任务），调整坍缩方向。

**"记住下次不要这样做"不是反馈——"这次失败是因为 deploy skill 没有处理多节点场景，已在 skill 中补充"才是反馈。** 前者是扁平的文本碎片，后者是结构化的系统改进。

# 第六部分：工具映射与 Gap 分析

## 第十八章：以 Claude Code 为例

理论再漂亮，最终要映射到现有工具上。以 Claude Code 为例，它提供了一套完整的机制栈：

- **CLAUDE.md** ——层级化的上下文文件，对应 L1 知识分层
- **Skills** ——按需加载的能力封装，对应定理一的选择性加载
- **MCP** ——外部工具连接，对应定理二的快变状态查询
- **Hooks** ——确定性拦截点，对应 L3 的自动验证层
- **Subagents** ——隔离 context 的子 Agent，对应 L0 的独立坍缩
- **权限系统** ——工具级别的 allow/deny，对应定理三的风险分级

**做得好的部分：**

Conventions 层是最完整的。CLAUDE.md 提供软约束（prompt 级别），Hooks 提供硬约束（确定性代码）。快变状态的处理也很强——Skills 中的动态命令注入把 shell 命令的执行结果直接加入 prompt，这就是定理二"按需查询"的工程实现。表达阶段的自校正循环也越来越成熟——Agent 可以跑命令、看浏览器、读日志、自动迭代。

**做得差的部分——三个关键 Gap：**

### Gap 1：ADR 缺失——人机接口断裂

Claude Code 没有结构化的决策记录机制。"为什么选了 gRPC 而不是 REST"这种决策，在 session 结束后就消失了。Auto-memory 只记住片段，没有"为什么"、没有上下文、没有替代方案的对比。

这个 Gap 发生在人机边界上。它断裂的不是工具链中的某个环节，而是人类决策和 LLM 执行之间的桥梁。没有 ADR，LLM 拿到的是残缺的上下文——它知道做什么，但不知道为什么。

### Gap 2：反馈不归因——闭环断裂

Auto-memory 记住的是"别用 X 方法"，但不会告诉你这条教训应该更新到哪里：改 Skill（流程有 bug）？改 CLAUDE.md（项目有特殊约束）？还是改 MCP 查询（环境已变化）？

没有归因，反馈只是扁平的文本碎片，无法驱动系统性改进。每条"教训"都飘在空中，没有着陆点。

### Gap 3：权限是工具维度的，不是后果维度的

权限系统基于工具类型和命令模式（"允许执行所有 git 命令"），但它无法理解语义风险。"修改一个 README"和"修改数据库 migration 文件"在权限系统看来可能都是文件写入——但后果天差地别。

L3 + 定理三要求验证强度与后果成正比。但当前工具只做到了工具粒度，不是后果粒度。

## 第十九章：角色定义的本质

一个重要的发现：**角色定义不是一个新的知识层——它是对现有层的遍历策略。**

当你给 Subagent 定义"你是安全审计工程师"时，你实际上做的是：

- 指定加载哪些 Conventions（安全相关的编码规范）
- 关注哪些 Task State（安全相关的 Issue）
- 提高 L3 验证的严格度（对权限、输入验证、加密等方面更挑剔）

角色不是一个新层。角色是一套策略，决定如何遍历现有的知识层、持久化栈和信任层次。这个理解让角色定义从"写一段人设"变成了"配置一个遍历策略"——后者可以被系统化、模板化、复用。

# 第七部分：交叉张力与工程权衡

## 第二十章：四条定律之间的张力

四条定律不是孤立的。它们的交叉组合产生了三组重要的工程权衡，任何实际系统都必须面对。

### L0 × L1：锐度-容量权衡

坍缩需要 prompt，知识需要 context，两者共享同一个有限空间。角色定义越详细（"你是一个专注于 Kubernetes 网络策略的、有 10 年经验的安全工程师，偏好声明式配置，反对过度抽象……"），坍缩越精确，但留给项目知识的空间就越少。

这个权衡没有通用的最优解，但有一个原则：**角色定义应该足够锐利以区分方向，但不要锐利到挤占关键的项目上下文。** 在实践中，50-200 词的角色定义 + 结构化的项目知识，通常优于 2000 词的超详细人设 + 残缺的项目信息。

### L0 × L2：坍缩的瞬时性

每次调用，坍缩都要重新发生。上一轮 session 中 Agent 已经完美地"成为"了一个架构师——下一轮，那个坍缩态消失了。你必须重新注入 prompt 让它再坍缩一次。角色不会被"记住"，只会被"重建"。

这就是为什么角色配置文件必须是持久化的、版本化的——它们不是一次性的 prompt，而是坍缩的模板，需要在每次调用时可靠地重现同一个智能实体。

### L0 × L3：坍缩不等于确定

即使 prompt 精确到"你是专注于 Kubernetes NetworkPolicy 审计的安全工程师"，输出仍然是概率性采样。坍缩只是移动了概率分布的中心，让安全相关的推理更可能出现——但它没有消除方差。Agent 仍然可能遗漏一个明显的 CVE，仍然可能写出"看起来安全但实际有漏洞"的代码。

这就是为什么即使有完美的角色定义，你仍然需要 L3 的验证层。坍缩和验证是互补的，不是替代的。坍缩提高了基准线（大部分输出更好），验证捕获尾部风险（偶尔的严重错误）。

# 第八部分：工具设计方向

## 第二十一章：两个最高优先级的切入点

从整条管线的 Gap 分析中，有两个切入点的 ROI 最高：

### 切入点一：人机边界的输入质量（解决 Gap 1）

在人机边界上建立一道结构化的"质量门"：

- **PRD 模板 + 完整性检查。** 不是通用的 PRD 模板，而是按照 L1 知识分层设计的模板——每个字段都对应 LLM 在表达阶段需要的一个信息维度。完整性检查自动检测缺失的字段。
- **ADR 生成辅助。** 在决策发生时（代码讨论、架构会议、PR 评论），自动提取决策要素，生成 ADR 草稿供人类确认。降低编码决策的成本——不是让人写更多文档，而是让文档从决策过程中自然产生。
- **任务拆分验证。** 检查每个任务是否在 LLM 单次调用的能力范围内，是否有足够的上下文支持独立执行。

### 切入点二：反馈归因的闭环（解决 Gap 2）

在验证失败时，建立结构化的归因流程：

- **失败分类。** 自动或半自动地判断失败原因属于哪一类：Skill bug、知识缺失、状态过期、角色不匹配。
- **定向更新。** 根据归因结果，自动生成对应知识层的更新建议——修改 Skill 文件、补充 ADR 条目、刷新环境配置、调整角色定义。
- **闭环验证。** 更新后，自动重跑失败的任务，验证修复是否生效。

这两个切入点的共同特征是：它们解决的不是"LLM 更聪明"的问题，而是"LLM 拿到更好的输入"和"LLM 的错误驱动系统性改进"的问题。

## 第二十二章：远期的后果比例验证（Gap 3）

Gap 3（权限是工具维度而非后果维度的）是一个更难解决的问题，因为它需要语义理解——判断一个文件修改的后果严重度，本身就是一个需要 LLM 能力的任务。

一个可能的方向是：在表达阶段的退出点（Agent 声称"完成"时），用一个独立的"风险评估 Agent"扫描所有变更，对每个变更标注后果级别，然后路由到对应的验证节点。这个 Agent 本身也是概率性的（L3），但它提供的是"分级"而非"判断"——即使分级有误，最多是验证强度不完全匹配，不会导致完全跳过验证。

# 结语

## 回到最初的问题

Agent 开发中，那些散落的 PRD、架构文档、部署流程、测试环境信息，和 Agent 到底是什么关系？

现在我们可以给出一个完整的回答：

**它们是两层理论在工程层面的具体投射。** PRD 和架构文档是人类理解和决策阶段的编码产出，是跨过人机边界的知识载体。部署流程和 CI/CD 定义是 Skill（可复用的能力封装）。测试环境信息是 Live State（需要实时查询的快变数据）。代码规范和安全红线是 Conventions（L3 的软硬约束）。角色定义是 L0 的坍缩方向。反馈记录是 L2 × L3 的归因产物。

每一层的存在都不是"组织整齐比较好"，而是"不这样做会违反某条定律"。

## 四个问题

整套架构可以浓缩为四个问题：

- **L0：让它成为什么？** → 角色定义、多 Agent、坍缩隔离
- **L1：喂它什么？** → 知识分层、选择性加载、人机边界的质量门
- **L2：留住什么？** → 频率匹配持久化、反馈归因闭环
- **L3：信任什么？** → 分级验证、后果比例升级

但在这四个问题之前，还有一个更基本的前提：

**人类先做好理解和决策，把成果编码为 LLM 可消费的知识产物。** 这不是工具能替你做的——这是人类不可替代的价值所在。

## 最终框架

```
认知模型层：   理解 → 决策 →|人机边界|→ 表达（含自校正循环） → 验证（独立坍缩态）
                                |            ↑                         |
                                |            |                         |
物理定律层：              L0/L1/L2/L3  ←  反馈归因（L2×L3）  ←————————┘
                                |
知识产物：        PRD · ADR · 约束清单 · 模块定义 · 任务清单
```

两层理论。一条边界。一个闭环。

不是偏好。不是经验。是推导。
